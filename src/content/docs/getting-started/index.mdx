This walks through the **TypeScript reference implementation**, `openbody-ts`: validate an
OpenBody record against the published JSON Schema, then run it through the
canonical-normalization pipeline (`conformance/EQUIVALENCE.md` — which the reference
implementation implements as the conformance suite's equivalence oracle). A reference
implementation is *one* implementation, not normative —
[`SPEC.md`](/specification/spec/) is the source of truth.

<Aside type="note" title="Pre-release tooling — not yet on npm">
The package is `@openbody/openbody-ts` (v0.1.0, tracking the current OpenBody
draft). It is **not yet published to npm**, so `npm install @openbody/openbody-ts` will not
work today. While the package is pre-release, the working setup is a **local clone**, as
below (a git-URL install — `npm install github:openbody/openbody-ts` — becomes an option
once the repository is public). Install steps will firm up at the public draft release.
</Aside>

## Install

<Steps>

1. Clone the standard and the reference implementation as sibling checkouts. The vector
   runner and validator read the standard (schema + vectors) from `../openbody` by default
   (override with `OPENBODY_STANDARD`).

   ```bash
   git clone https://github.com/openbody/openbody.git
   git clone https://github.com/openbody/openbody-ts.git
   cd openbody-ts
   npm install
   ```

2. Confirm the toolchain is green:

   ```bash
   npm test   # typecheck + lint + vitest (including lossless number checks)
   ```

</Steps>

## Validate a record

`validate(record)` checks a record against the OpenBody JSON Schema (Draft 2020-12,
SPEC §§4–7). Validate **wire** records — not the canonical form the equivalence method
produces, which is a *comparison* artifact, not the binding.

```ts
import { validate } from "@openbody/openbody-ts";

// A minimal reps-scored WorkUnit — the one required-tier element of Training core (§3.3).
const workUnit = {
  id: "wu-squat-1",
  recordType: "WorkUnit",
  subject: "subj-001",
  exerciseRef: "squat.barbell.high-bar",
  scoring: "reps",
  prescription: { reps: 5, load: { value: 100, unit: "kg", basis: "marked_weight" } },
};

const result = validate(workUnit);
console.log(result); // → valid / list of schema errors
```

<Aside type="caution" title="Schema validates structure, not every rule">
The JSON Schema validates **structure**. Several context-dependent rules are enforced by a
conforming implementation, not the schema — e.g. a `Session` carries at most one of
`blocks|exercises|workUnits` (§5.3); `scoring`↔metric agreement (§5.5); `Load.unit`
conditionality (§5.12); `algorithm` required when `method` is `algorithm` (§7.4). See the
[JSON Schema page](/specification/schema/) and the schema's own `NOTE`.
</Aside>

## Normalize & compare

`normalizeDocument(doc)` runs the [canonical-normalization pipeline](/concepts/canonicalization/)
(`conformance/EQUIVALENCE.md`)
— number → lowest-terms fixed-point, unit canonicalization, scalar→`Target` expansion,
`ExerciseRef` fold, `sets` expansion, deterministic id assignment, flatten + `partOf`,
`status` default, then RFC 8785 serialization — producing a sorted set of canonical record
byte strings. `equivalent(a, b)` is true iff two documents normalize to the same set.

```ts
import { equivalent } from "@openbody/openbody-ts";

// The nested document and the flat + partOf encoding of one structure are equivalent (§7.2).
console.log(equivalent(nestedDoc, flatDoc)); // → true
```

<Aside type="tip" title="Lossless numbers">
For full canonicalization fidelity (`EQUIVALENCE.md` step 1), feed documents through `parseLossless` (or raw JSON text) rather
than `JSON.parse`: numbers are parsed losslessly from their decimal text, never via
`float64`, so high-precision decimals and integers above 2⁵³ canonicalize to their exact
value. `npm test` proves it (the lossless-number checks run under vitest).
</Aside>
