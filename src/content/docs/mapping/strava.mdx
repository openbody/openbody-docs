**Source:** a Strava activity + its streams. **Pillars:** A + B. **Mapper:**
[`mapStrava`](https://github.com/openbody/openbody-ts/blob/main/src/mappers/strava.ts) —
`mapStrava({ activity, streams }) → { records, warnings }`. This is the cross-pillar shape end to end:
telemetry becomes Pillar A `Measurement`s, and a Pillar B `Session` **references** them via
`measuredBy` rather than embedding them (§1.3).

## Structural correspondence

| Strava | OpenBody |
|---|---|
| `streams.heartrate` / `watts` / `cadence` | single-channel `sampleArray` `Measurement` (`heart_rate` `/min`, `power` `W`, `cadence` `/min`) |
| `streams.latlng` + `altitude` | one **multi-channel** `location` `sampleArray` (channels `lat`/`lon`/`alt`) |
| `streams.time` | the shared `offsets` array (irregular sampling) |
| `average_heartrate` / `max_heartrate` | interval `quantity` aggregates (`heart_rate_mean`/`_max`), with a `derivedFrom` link to the HR stream **when that stream was fetched** |
| `sport_type` (`"Run"`, `"Ride"`, …) | a registry discipline token (`"running"`, `"cycling"`, …); unknown types round-trip namespaced as `strava:<type>` (§4.4 ladder) |
| the activity itself | `Session` (+ a `continuous` `WorkUnit`) with `measuredBy` links to every telemetry `Measurement` |
| `device_name` | `provenance.device` (`{ model }` — the display string names no manufacturer separately, so none is fabricated); `sourceApp: "strava"` |
| `name` (activity title) | `Session.name`, when present |


<Aside type="caution" title="derivedFrom ⇒ provenance.algorithm (recommended)">
A `derivedFrom` link makes `provenance.algorithm` **recommended** (§7.4, relaxed in v0.3.1
from *required*). Strava doesn't publish its aggregation algorithm, so the mapper supplies a
best-effort `algorithm.name` (`"strava-summary"`). When you genuinely can't name the
algorithm, omit it — the relaxed rule exists precisely so a faithful mapper never has to
drop the lineage link or fabricate a value. The link itself is **conditional**: the HR
aggregates carry `derivedFrom` only when the HR stream was actually fetched — a summary-only
activity (no streams) gets the aggregates without a dangling link.
</Aside>

## Notes

- **Input shape**: `mapStrava` consumes the Strava API's *activity* + *streams* responses —
  it doesn't fetch them itself. The streams **must include `time`** (the `sampleArray`
  offsets can't be computed without it); the mapper throws a clear error otherwise.
- **Disciplines**: `sport_type` maps to registry discipline tokens (`"Run"` → `"running"`,
  `"Ride"` → `"cycling"`, `"WeightTraining"` → `"strength"`, …); anything unmapped
  round-trips as a namespaced `strava:<type>` token.
- **`continuous` scoring** may carry any of `distance`, `time`, `energy` and requires no
  single metric — the endurance shape works end to end.
- **`endTime`** is `start_date + elapsed_time` (2600 s → `07:13:20Z`); `moving_time` (2520 s)
  is the `WorkUnit`'s recorded `time`.
- **`peerSensor`** (a parallel co-stream, e.g. two HR straps) is distinct from
  `sameActivityAs` (whole-activity dedup across sources) — see the
  [data model](/concepts/data-model/#everything-is-a-record-with-a-shared-envelope).
