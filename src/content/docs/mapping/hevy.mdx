**Source:** Hevy CSV export (one row per set). **Pillar:** B (strength). **Mapper:**
[`mapHevy`](https://github.com/openbody/openbody-ts/blob/main/src/mappers/hevy.ts) вЂ”
`mapHevy(csv) в†’ { records, warnings }`, one `Session` per workout.

## Structural correspondence

| Hevy CSV | OpenBody |
|---|---|
| a workout (rows sharing `title` + `start_time`) | `Session` (`disciplines: ["strength"]`, `name` в†ђ `title`, `notes` в†ђ `description`, `clientRecordId` в†ђ the natural key) |
| consecutive rows for one `exercise_title` | `Exercise` (`exerciseRef`: canonical registry `id` where the crosswalk resolves the title, plus the lossless `opaque` original) |
| one set row | `WorkUnit` (`scoring` from which columns are present) |
| `set_type` (`normalпЅњwarmupпЅњdropпЅњfailure`) | `WorkUnit.setRole` (`workingпЅњwarmupпЅњdropпЅњfailure`) |
| `weight_kg` | `performance.load` (`unit: "kg"`) |
| `reps` / `distance_km` / `duration_seconds` | `performance.reps` / `.distance` / `.time` |
| `rpe` | `performance.effortLoad` (`{ kind: "internal", method: "RPE", value }`) |
| `superset_id` (non-empty) | a `Block` with `grouping: "superset"` |

## Input (one set)

```csv
"title","start_time","end_time",...,"exercise_title","superset_id",...,"set_type","weight_kg","reps",...,"rpe"
"Morning workout","22 Dec 2025, 08:00","22 Dec 2025, 08:37",...,"Pull Up (Assisted)",,...,"normal",21,10,,0,8.5
```

<Aside type="tip" title="Assisted machines в†’ Load.basis: assist">
"Pull Up (Assisted)" is detected by name and emits `basis: "assist"` rather than pretending
the weight is added resistance вЂ” the machine's assist load round-trips honestly (В§5.12).
Plain machine/weighted lifts use `basis: "marked_weight"`.
</Aside>

## Notes & edge cases

- **Stable ids.** The export has no workout id of its own, so the natural key
  (`title|start_time`) becomes `clientRecordId` (В§7.1) and a **content hash** of it the
  record id вЂ” positional numbering would renumber everything when one more workout is
  exported, defeating dedup.
- **Workout title** maps to the first-class `name` (added in v0.3); the first Hevy mapper
  silently dropped it вЂ” a real losslessness bug that dogfooding surfaced.
- **Supersets force all-blocks.** A `Session` carries **at most one** of
  `blocksпЅњexercisesпЅњworkUnits` (В§5.3). If *any* row has a `superset_id`, the mapper emits
  everything under `blocks[]`, wrapping standalone exercises in singleton `Block`s and
  grouping superset mates in a `Block` with `grouping: "superset"`.
- **Scoring** is chosen per set: `reps` if present, else `distance`, else `time` вЂ” so plank
  and cardio rows map cleanly alongside strength sets.
- **Exercise identity**: names that the [registry](/registry/exercises/) crosswalk covers
  resolve to canonical ids **alongside** the `opaque` original (the fixture's
  `"Leg Press (Machine)"` becomes `{ "id": "leg-press.machine", "opaque": "Leg Press (Machine)" }`);
  anything unresolved вЂ” like `"Pull Up (Assisted)"` above вЂ” keeps the `opaque` floor.
  Lossless either way, per the
  [matching ladder](/concepts/exercise-identity/#the-matching-ladder).

<Aside type="note">
Timestamps are normalized to RFC 3339; the Hevy export carries no UTC offset, so the mapper
stamps the wall-clock time with the optional `utcOffset` option (default `"Z"`), never the
machine's local timezone вЂ” the same bytes map to the same records everywhere. A `Z` here is
the mapper's default assumption, not source-provided offset information.
</Aside>
