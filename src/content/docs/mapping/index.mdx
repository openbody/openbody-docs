Bidirectional mapping to and from incumbent schemas is OpenBody's **primary success
measure** and its adoption wedge: the standard is a **universal adapter**. These guides are
built from the real mappers in [`openbody-ts/src/mappers`](https://github.com/openbody/openbody-ts/tree/main/src/mappers)
— pure `input → { records, warnings }` functions, round-trip tested (`npm test`, 335 tests).
Structurally unusable input throws a typed `MapperInputError`; anything merely degraded,
skipped, or defaulted is reported on the returned `warnings` channel instead. The mappers
resolve source exercise names to canonical [registry](/registry/exercises/) ids where the
registry covers them, falling back to the lossless `opaque` form otherwise.

## Supported formats

All mappers are importable from the `openbody-ts` package root and run in both node and the
browser. The [in-browser converter](/tools/convert/) wires a subset of them to a drag-and-drop
page.

| Format | Direction | Works in | Caveats |
|---|---|---|---|
| [Hevy](/mapping/hevy/) (workout CSV) | import | library + [browser tool](/tools/convert/) | |
| [Strong](/mapping/strong/) (workout CSV) | import | library + [browser tool](/tools/convert/) | `,`- or `;`-delimited exports both parse |
| [Strava](/mapping/strava/) (activity + streams JSON) | import | library + [browser tool](/tools/convert/) | the library maps the API's activity+streams responses (streams must include `time`); the browser tool reads the bulk-export `activities.csv` instead |
| [Apple Health / Health Connect](/mapping/apple-health/) (`export.xml`) | import | library + [browser tool](/tools/convert/) | |
| [FIT](/mapping/fit/) (recorded activity or structured workout) | import | library | takes **pre-decoded** FIT messages — the caller decodes the binary ([why](/mapping/fit/)) |
| [GPX](/mapping/gpx/) (GPS track) | import | library | untimed tracks degrade honestly (no fabricated timing) |
| [TCX](/mapping/tcx/) (Garmin Training Center XML) | import | library | |
| [Fitbit](/mapping/fitbit/) (Google Takeout folder) | import | library | takes a list of `{ name, text }` files; timestamps need a `utcOffset` |
| [Concept2](/mapping/concept2/) (Logbook season CSV) | import | library | one row per workout — per-stroke detail isn't in this export |
| [theCrag](/mapping/thecrag/) (logbook CSV) | import | library | |
| [Strong](/mapping/to-strong/) (CSV out — also imports into Hevy) | **export** | library + [browser tool](/tools/convert/) | best-effort by design: losses reported machine-readably ([contract](/mapping/to-strong/)) |

The browser tool currently accepts **Hevy, Strong, Apple Health, and Strava** files; the
other formats run through the library. CSV timestamps carry no UTC offset at the source, so
the CSV-based mappers take an optional `utcOffset` and are timezone-independent — the same
bytes map to the same records on any machine.

<Aside type="note" title="The lossless guarantee is directional (§10)">
- *incumbent → OpenBody* is **lossless** — OpenBody is the superset; anything lacking a core
  home is preserved via extensions or the opaque fallback.
- *source → OpenBody → source* (a **source-anchored round-trip**) is **lossless**.
- Emitting OpenBody's full expressiveness *into a less-expressive target* (a %1RM target or
  a superset into a schema with no field for it) is necessarily best-effort and **bounded by
  that target** — a limit of the target, not a defect of OpenBody.
</Aside>

## Mapping principles (§10.1)

- **Canonical-plus-residue.** Map each source construct to its canonical OpenBody
  equivalent; carry anything with no canonical home in **namespaced extensions** or, for
  identity, the **opaque** fallback. Nothing is dropped.
- **Lazy terminology.** Source type tokens are preserved source-namespaced on ingest and
  resolved to external codes (LOINC / SNOMED / UCUM / MET) only on output.
- **Provenance preserved.** Source platform, app, and device become registry-backed
  provenance tokens; the source's own key becomes `clientRecordId`.
- **Round-trip.** Source → canonical → source **MUST** preserve meaning under the
  [§8.3 equivalence rule](/concepts/canonicalization/).

<Aside type="tip" title="What fit cleanly, and what to watch">
Real-data dogfooding found **no model gaps** for strength (Hevy/Strong), telemetry
(Strava), Pillar A (Apple Health), or FIT's prescription side (`workout`/`workout_step`) —
the first mapper here to map planned data rather than performed. Things to keep in mind,
surfaced by the mappers: every app has a workout **title** (now the v0.3 `name`);
`derivedFrom` ⇒ `provenance.algorithm` is *recommended* and incumbents rarely publish it; a
`Session` carries **at most one** container kind (§5.3), so a mix of supersets + standalone
exercises (Hevy/Strong) or a mix of repeated and one-off steps (FIT) maps to **all-blocks**;
and `WorkUnit` has no `name` field, so a per-step label (FIT's `wkt_step_name`) lives on the
wrapping `Block` instead.
</Aside>
