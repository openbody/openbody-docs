This walks through the **TypeScript reference implementation**, `openbody-ts`: validate an
OpenBody record against the published JSON Schema, then run it through the ¬ß8.3
canonical-normalization pipeline. A reference implementation is *one* implementation, not
normative ‚Äî [`SPEC.md`](/specification/spec/) is the source of truth.

<Aside type="note" title="Pre-release tooling ‚Äî not yet on npm">
The package is `@openbody/openbody-ts` (early, v0.1.0, tracking the current OpenBody
draft). It is **not yet published to npm**, so `npm install @openbody/openbody-ts` will not
work today. While the package is pre-release, the working setup is a **local clone**, as
below (a git-URL install ‚Äî `npm install github:openbody/openbody-ts` ‚Äî becomes an option
once the repository is public). Install steps will firm up at the public draft release.
</Aside>

## Install

<Steps>

1. Clone the standard and the reference implementation as sibling checkouts. The vector
   runner and validator read the standard (schema + vectors) from `../openbody` by default
   (override with `OPENBODY_STANDARD`).

   ```bash
   git clone https://github.com/openbody/openbody.git
   git clone https://github.com/openbody/openbody-ts.git
   cd openbody-ts
   npm install
   ```

2. Confirm the toolchain is green:

   ```bash
   npm test   # typecheck + lossless number checks + vectors + mapper round-trips
   ```

</Steps>

## Validate a record

`validate(record)` checks a record against the OpenBody JSON Schema (Draft 2020-12,
SPEC ¬ß¬ß4‚Äì7). Validate **wire** records ‚Äî not the ¬ß8.3 canonical form, which is a
*comparison* artifact, not the binding.

```ts
import { validate } from "@openbody/openbody-ts";

// A minimal reps-scored WorkUnit ‚Äî the one required-tier element of Training core (¬ß3.3).
const workUnit = {
  id: "wu-squat-1",
  recordType: "WorkUnit",
  subject: "subj-001",
  exerciseRef: "squat.barbell.high-bar",
  scoring: "reps",
  prescription: { reps: 5, load: { value: 100, unit: "kg", basis: "marked_weight" } },
};

const result = validate(workUnit);
console.log(result); // ‚Üí valid / list of schema errors
```

<Aside type="caution" title="Schema validates structure, not every rule">
The JSON Schema validates **structure**. Several context-dependent rules are enforced by a
conforming implementation, not the schema ‚Äî e.g. a `Session` carries at most one of
`blocks|exercises|workUnits` (¬ß5.3); `scoring`‚Üîmetric agreement (¬ß5.5); `Load.unit`
conditionality (¬ß5.12); `algorithm` required when `method` is `algorithm` (¬ß7.4). See the
[JSON Schema page](/specification/schema/) and the schema's own `NOTE`.
</Aside>

## Normalize & compare

`normalizeDocument(doc)` runs the [¬ß8.3 canonical-normalization pipeline](/concepts/canonicalization/)
‚Äî number ‚Üí lowest-terms fixed-point, unit canonicalization, scalar‚Üí`Target` expansion,
`ExerciseRef` fold, `sets` expansion, deterministic id assignment, flatten + `partOf`,
`status` default, then RFC 8785 serialization ‚Äî producing a sorted set of canonical record
byte strings. `equivalent(a, b)` is true iff two documents normalize to the same set.

```ts
import { equivalent } from "@openbody/openbody-ts";

// The nested document and the flat + partOf encoding of one structure are equivalent (¬ß7.2).
console.log(equivalent(nestedDoc, flatDoc)); // ‚Üí true
```

<Aside type="tip" title="Lossless numbers">
For full ¬ß8.3 fidelity, feed documents through `parseLossless` (or raw JSON text) rather
than `JSON.parse`: numbers are parsed losslessly from their decimal text, never via
`float64`, so high-precision decimals and integers above 2‚Åµ¬≥ canonicalize to their exact
value. `npm run lossless` proves it.
</Aside>
